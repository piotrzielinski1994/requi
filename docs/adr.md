# Architectural Decisions — requi

Append-only log of architectural and design decisions made during development.

## Format

Each entry follows this structure:

| Date | Decision | Rationale |
|------|----------|-----------|
| {YYYY-MM-DD} | {What was decided} | {Why this choice was made} |

## Decisions

| Date | Decision | Rationale |
|------|----------|-----------|
| 2026-06-18 | Desktop shell via Tauri (not Electron) | Smaller bundle, native webview, Rust backend; core platform choice, expensive to swap |
| 2026-06-18 | Adopt TanStack ecosystem (Router/Query/Table/Form) | Single coherent stack; permeates whole frontend architecture |
| 2026-06-18 | Keybindings via @tanstack/hotkeys despite alpha status | Official TanStack lib chosen over stable react-hotkeys-hook; alpha = API-churn risk, isolated behind command-palette component |
| 2026-06-19 | Workspace on-disk format = directory-tree mirror (folder=dir, request=`*.req.json`, per-folder `folder.json`, `requi.workspace.json` manifest) | Git-friendly and diffable, scales to large collections, matches the file-based ethos (Bruno-like). It is the public file contract - costly to reverse once users have workspaces on disk. Alternatives (single JSON file; file + sidecar) lost on diffability/scale |
| 2026-06-19 | Config inheritance model: each folder/request carries an optional `ConfigScope`; a request resolves by folding root->leaf, child overrides parent (per-key merge for variables/headers/params, nearest-defined-wins for auth/scripts/timeout), `undefined`/`{inherit}` = inherit, every resolved value keeps provenance | Core semantic contract of the product; whole-object-vs-field-merge for auth was genuinely debated (chose nearest whole-object). Provenance idea borrowed from the pricing archetype's `ComponentBreakdown`; full archetype machinery rejected as overkill |
| 2026-06-19 | Keyboard shortcuts = static action registry (`SHORTCUT_ACTIONS`, defaults baked in) + a sparse per-installation override map (`settings.shortcuts`, action-id -> hotkey); effective binding = `resolveShortcuts(overrides)` overlaying defaults. Only currently-wired actions are in the registry; dead buttons (Send/New-request) excluded. Block+warn on conflicts | Registry-as-data keeps the action list declarative/testable and the override map small + forward-compatible (unknown ids/invalid hotkeys dropped on merge). Alternative (store full binding per action) rejected: defaults would be frozen at first-save and couldn't evolve. Conflict policy: block+warn chosen over silent-override/allow-duplicates to avoid silently breaking an existing binding |
| 2026-06-19 | Settings is an in-app tab in the workspace shell (state in `WorkspaceProvider`), NOT a `/settings` router route | The route model caused two bugs in manual testing: `Alt`+arrows walked browser history (created by `navigate`) instead of cycling request tabs, and navigating away unmounted `WorkspaceProvider`, resetting open tabs + expanded folders. Removing the route eliminates both at the root (no history entry, no unmount). Alternative (keep route + `preventDefault` the history keys + hoist provider) rejected: WebView native back/forward isn't reliably suppressible and it leaves the unmount-on-nav fragility |
| 2026-06-19 | Default tab-switch shortcuts = `Ctrl+Tab` / `Ctrl+Shift+Tab` (not `Alt`+arrows) | The user's Karabiner-Elements config remaps `Option`+arrows and `Cmd+W` at the OS level before the WebView sees them, so those defaults silently never reached the app. `Ctrl+Tab`/`Ctrl+Shift+Tab` are the only tab-switch combos Karabiner leaves intact. Keys are user-rebindable anyway; this just picks defaults that work out-of-the-box on the primary dev machine |
| 2026-06-19 | `new-request` opens an in-memory ephemeral draft (id `draft-<n>`), NOT a disk write | The disk write/edit model doesn't exist yet (workspace is read-only on disk today); building it is a separate feature. Drafts let the shortcut + `+` button do something real now, merged into `requestsById` so all tab machinery works. Trade-off accepted: drafts vanish on reload until persistence lands |
| 2026-06-19 | `open-workspace` via a `FolderPicker` port + `tauri-plugin-dialog` | Adds the first native-dialog dependency. Port keeps it testable (noop/fake) and browser-safe. Picker threaded as a prop (loader->layout->main) rather than a new context, matching the existing `fs` prop pattern |
| 2026-06-19 | Command palette built on shadcn `command` (cmdk) + `dialog`, sourced from the shortcut registry. Palette is itself a registry action (`open-command-palette`, default `Mod+K`); the command list = `SHORTCUT_ACTIONS` minus itself, run via `Main`'s shared handler map | Adds `cmdk` + `@radix-ui/react-dialog` deps. cmdk gives type-to-filter + arrow-nav + a11y natively (the requested behavior) and is the engine of the shadcn command component the project already standardizes on - reinventing it as a hand-built overlay was rejected as ifology. Sourcing commands from the existing registry (vs a separate command list) avoids a second source of truth that would drift from shortcuts; the registry already carries name/description/binding per action. The ADR predicted this: `@tanstack/hotkeys` was chosen partly because it would be "isolated behind a command-palette component" |
| 2026-06-19 | Request body editor = CodeMirror 6 (`@uiw/react-codemirror` + `@codemirror/lang-json` + `closeBrackets`), edits stored in-memory only (`bodyOverrides` map in `WorkspaceProvider`), no disk write | CM6 chosen over Monaco (heavy, worker setup fights Vite/Tauri webview) and textarea+Prism (hand-built bracket logic, overlay-only highlight); it's light, themeable to the oklch tokens, and ships JSON highlight + bracket-closing. In-memory-only matches the current app state (disk write/save is a separate unbuilt feature; drafts are already in-memory) and keeps the feature YAGNI-scoped to "a comfortable editor". JSON+plaintext only (no Content-Type-based grammar detection) since every body is JSON today |
| 2026-06-19 | Body editor theme = JetBrains Darcula via hand-composed `EditorView.theme` + `HighlightStyle` (NOT `@uiw/codemirror-themes`/`createTheme`); error checking = JSON syntax linting (`jsonParseLinter`), not JSON-Schema type validation | User asked for "IntelliJ Dracula like the screenshot" - the screenshot tokens are actually Darcula (default IntelliJ dark: olive strings, steel-blue numbers, orange keywords), so we matched Darcula, not the vivid classic-Dracula palette. Built the theme from `@codemirror/language` + `@lezer/highlight` (both already peers of the wrapper) to avoid a new dep, since `createTheme` isn't re-exported. The screenshot's "Expected string" errors are JSON-Schema type checks, which need a schema a freeform request body doesn't have; scoped error-checking to parse/syntax linting (red underline on malformed JSON) and deferred schema validation as a separate future feature |
