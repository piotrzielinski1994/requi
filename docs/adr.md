# Architectural Decisions — requi

Append-only log of architectural and design decisions made during development.

## Format

Each entry follows this structure:

| Date | Decision | Rationale |
|------|----------|-----------|
| {YYYY-MM-DD} | {What was decided} | {Why this choice was made} |

## Decisions

| Date | Decision | Rationale |
|------|----------|-----------|
| 2026-06-18 | Desktop shell via Tauri (not Electron) | Smaller bundle, native webview, Rust backend; core platform choice, expensive to swap |
| 2026-06-18 | Adopt TanStack ecosystem (Router/Query/Table/Form) | Single coherent stack; permeates whole frontend architecture |
| 2026-06-18 | Keybindings via @tanstack/hotkeys despite alpha status | Official TanStack lib chosen over stable react-hotkeys-hook; alpha = API-churn risk, isolated behind command-palette component |
| 2026-06-19 | Workspace on-disk format = directory-tree mirror (folder=dir, request=`*.req.json`, per-folder `folder.json`, `requi.workspace.json` manifest) | Git-friendly and diffable, scales to large collections, matches the file-based ethos (Bruno-like). It is the public file contract - costly to reverse once users have workspaces on disk. Alternatives (single JSON file; file + sidecar) lost on diffability/scale |
| 2026-06-19 | Config inheritance model: each folder/request carries an optional `ConfigScope`; a request resolves by folding root->leaf, child overrides parent (per-key merge for variables/headers/params, nearest-defined-wins for auth/scripts/timeout), `undefined`/`{inherit}` = inherit, every resolved value keeps provenance | Core semantic contract of the product; whole-object-vs-field-merge for auth was genuinely debated (chose nearest whole-object). Provenance idea borrowed from the pricing archetype's `ComponentBreakdown`; full archetype machinery rejected as overkill |
| 2026-06-19 | Keyboard shortcuts = static action registry (`SHORTCUT_ACTIONS`, defaults baked in) + a sparse per-installation override map (`settings.shortcuts`, action-id -> hotkey); effective binding = `resolveShortcuts(overrides)` overlaying defaults. Only currently-wired actions are in the registry; dead buttons (Send/New-request) excluded. Block+warn on conflicts | Registry-as-data keeps the action list declarative/testable and the override map small + forward-compatible (unknown ids/invalid hotkeys dropped on merge). Alternative (store full binding per action) rejected: defaults would be frozen at first-save and couldn't evolve. Conflict policy: block+warn chosen over silent-override/allow-duplicates to avoid silently breaking an existing binding |
| 2026-06-19 | Settings is an in-app tab in the workspace shell (state in `WorkspaceProvider`), NOT a `/settings` router route | The route model caused two bugs in manual testing: `Alt`+arrows walked browser history (created by `navigate`) instead of cycling request tabs, and navigating away unmounted `WorkspaceProvider`, resetting open tabs + expanded folders. Removing the route eliminates both at the root (no history entry, no unmount). Alternative (keep route + `preventDefault` the history keys + hoist provider) rejected: WebView native back/forward isn't reliably suppressible and it leaves the unmount-on-nav fragility |
| 2026-06-19 | Default tab-switch shortcuts = `Ctrl+Tab` / `Ctrl+Shift+Tab` (not `Alt`+arrows) | The user's Karabiner-Elements config remaps `Option`+arrows and `Cmd+W` at the OS level before the WebView sees them, so those defaults silently never reached the app. `Ctrl+Tab`/`Ctrl+Shift+Tab` are the only tab-switch combos Karabiner leaves intact. Keys are user-rebindable anyway; this just picks defaults that work out-of-the-box on the primary dev machine |
| 2026-06-19 | `new-request` opens an in-memory ephemeral draft (id `draft-<n>`), NOT a disk write | The disk write/edit model doesn't exist yet (workspace is read-only on disk today); building it is a separate feature. Drafts let the shortcut + `+` button do something real now, merged into `requestsById` so all tab machinery works. Trade-off accepted: drafts vanish on reload until persistence lands |
| 2026-06-19 | `open-workspace` via a `FolderPicker` port + `tauri-plugin-dialog` | Adds the first native-dialog dependency. Port keeps it testable (noop/fake) and browser-safe. Picker threaded as a prop (loader->layout->main) rather than a new context, matching the existing `fs` prop pattern |
| 2026-06-19 | Command palette built on shadcn `command` (cmdk) + `dialog`, sourced from the shortcut registry. Palette is itself a registry action (`open-command-palette`, default `Mod+K`); the command list = `SHORTCUT_ACTIONS` minus itself, run via `Main`'s shared handler map | Adds `cmdk` + `@radix-ui/react-dialog` deps. cmdk gives type-to-filter + arrow-nav + a11y natively (the requested behavior) and is the engine of the shadcn command component the project already standardizes on - reinventing it as a hand-built overlay was rejected as ifology. Sourcing commands from the existing registry (vs a separate command list) avoids a second source of truth that would drift from shortcuts; the registry already carries name/description/binding per action. The ADR predicted this: `@tanstack/hotkeys` was chosen partly because it would be "isolated behind a command-palette component" |
| 2026-06-19 | Request body editor = CodeMirror 6 (`@uiw/react-codemirror` + `@codemirror/lang-json` + `closeBrackets`), edits stored in-memory only (`bodyOverrides` map in `WorkspaceProvider`), no disk write | CM6 chosen over Monaco (heavy, worker setup fights Vite/Tauri webview) and textarea+Prism (hand-built bracket logic, overlay-only highlight); it's light, themeable to the oklch tokens, and ships JSON highlight + bracket-closing. In-memory-only matches the current app state (disk write/save is a separate unbuilt feature; drafts are already in-memory) and keeps the feature YAGNI-scoped to "a comfortable editor". JSON+plaintext only (no Content-Type-based grammar detection) since every body is JSON today |
| 2026-06-19 | Body editor theme = JetBrains Darcula via hand-composed `EditorView.theme` + `HighlightStyle` (NOT `@uiw/codemirror-themes`/`createTheme`); error checking = JSON syntax linting (`jsonParseLinter`), not JSON-Schema type validation | User asked for "IntelliJ Dracula like the screenshot" - the screenshot tokens are actually Darcula (default IntelliJ dark: olive strings, steel-blue numbers, orange keywords), so we matched Darcula, not the vivid classic-Dracula palette. Built the theme from `@codemirror/language` + `@lezer/highlight` (both already peers of the wrapper) to avoid a new dep, since `createTheme` isn't re-exported. The screenshot's "Expected string" errors are JSON-Schema type checks, which need a schema a freeform request body doesn't have; scoped error-checking to parse/syntax linting (red underline on malformed JSON) and deferred schema validation as a separate future feature |
| 2026-06-20 | Drag-and-drop via `@dnd-kit/*` (core+sortable+utilities), not native HTML5 DnD | Pointer + keyboard accessible, clean drop indicators, robust nested-tree reparenting. Native HTML5 reparent-with-drop-position in a WebView is fiddly and inaccessible. New dep accepted - tree+tab DnD is genuinely hard to do well by hand |
| 2026-06-20 | Sidebar moves WRITE TO DISK (new `WorkspaceFs.writeWorkspace` path), not in-memory-only | User decision: a moved request/folder must survive reload. This builds the first workspace writer (`serialize` was defined but unused). Reconcile diff is managed-files-only (regex-gated) + round-trip tested so it can't corrupt user files; in-memory remains the session source of truth and a failed write logs without rolling back (unlike body edits/drafts which stay in-memory - moves are the first persisted mutation) |
| 2026-06-20 | On-disk format gains an explicit per-node `order` field; `schemaVersion` 1->2 | Sibling reorder cannot otherwise persist - the format always re-sorted folders-first-then-alphabetical on load. Additive + tolerant: `serialize` preserves caller order and writes `order`, `deserialize` sorts by it with the old rule as fallback, so v1 workspaces still load. Public on-disk contract change (users have workspaces on disk), hence the version bump. Alternative (reparent-only, no reorder persistence) rejected as inconsistent UX |
| 2026-06-20 | Node ids stay STABLE through a move (no path-based id remap) | `resolveConfig` + all runtime lookups treat id as an opaque key (tree walked structurally); the path-shaped id is only a deserialize convention. Stable ids mean open tabs / selection / expanded-folder state survive a move for free, and `serialize` regenerates disk paths from structure on the next write. Avoided a fragile id-rewriting pass |
| 2026-06-20 | HTTP send goes through a Rust `send_http_request` command (`reqwest`, rustls TLS) behind an `HttpClient` port, NOT webview `fetch` or `@tauri-apps/plugin-http` | The webview is CORS-bound; arbitrary cross-origin requests fail. Routing through a Rust command bypasses CORS (standard Tauri pattern). Owning the command (vs plugin-http) keeps the request-building logic in tested TS and the Rust side a thin transport; rustls avoids a system OpenSSL build dependency. The port mirrors the existing `fs`/`folder-picker` seams so component/context tests inject a fake client and never hit the network |
| 2026-06-20 | URL/method/body edits unified into one `requestOverrides: Map<id, Partial<{url,method,body}>>` session map (generalizing the old `bodyOverrides: Map<id,string>`); send state in a parallel `responseStates: Map<id, ResponseState>` | Editing URL/method needs the same in-memory-override semantics body already had (per-id, cleared on close, reverts on reopen, no disk write). One map of partials beats three parallel maps; `setRequestBody` becomes a thin wrapper writing `{body}`, and the existing body-override test suite is the regression net proving the refactor is behavior-preserving. Disk persistence of edits remains a separate unbuilt feature |
| 2026-06-20 | Response filter = hand-rolled JSONPath subset (`$`, `.key`, `[index]`) returning an ADT result, NOT a JSONPath/JMESPath library | The requested behavior (narrow the shown body, matching the existing `$.data.items[0]` placeholder) needs only dot + index navigation; a full JSONPath engine (filters, wildcards, recursive descent) is YAGNI and a new dep. Total function (`{ok:true,text}` / `{ok:false}`, no throw) keeps the "no match" / non-JSON / empty-body paths as clean UI states rather than try/catch |
| 2026-06-20 | Environments live INSIDE folder/request `config` (new `config.environments: {envName: {var: val}}`), NOT in dedicated `environments/*` files; only `.env` is a dedicated file | User directive. Reuses the existing JSON config contract - `serialize`/`deserialize` already persist `config` opaquely, so no disk-format version bump and no new loader; the env-name list is derived from the tree, never stored. Diverges from Bruno (which uses `environments/<name>.bru`), but keeps one consistent config mechanism. `.env` stays a separate gitignored dotenv at the workspace root (the secrets path) |
| 2026-06-20 | Variable precedence: nearest scope wins (request > nearer folder > farther folder); WITHIN a scope a plain `variable` overrides that scope's active-environment block; `{{process.env.X}}` is a separate namespace | Bruno's published precedence puts folder/request above environment - chosen over "env overrides folder" so a folder-pinned value isn't silently changed by switching env. Plain-var-over-env-block (within a scope) makes `variables` act as defaults that an env can fill but not override locally (user directive). `process.env` separate namespace matches Bruno (`{{process.env.KEY}}` only) |
| 2026-06-20 | Recursive `{{var}}` interpolation with a cycle guard + unknown-token passthrough, hand-rolled in `lib/http/interpolate.ts` (replacing the single-pass `String.replace` in `buildHttpRequest`), extended to cover auth values + body | Bruno resolves variables-referencing-variables (e.g. an env var `token: {{process.env.JWT}}`); a single pass can't. Loop-until-stable with a per-token `visited` set terminates on `a<->b`/`a->a` cycles (token left verbatim) instead of hanging. Unknown tokens stay verbatim (matches prior behavior). Hand-rolled - no dep, total function |
| 2026-06-20 | In-UI editing of node config + `.env` (REVERSES the earlier "hand-edit on disk only" stance) | First in-app config WRITE. User asked to edit config/`.env` from the UI. Raw-JSON editor per node (folder + request) + a raw `.env` text editor, explicit Save (disabled while JSON invalid). Chosen over structured per-field forms (much larger; raw JSON ships now and covers vars/environments/headers/params/auth uniformly) and over auto-save (writes transient/invalid states). Node config reuses the existing `onTreeChange`/`writeWorkspace` path; `.env` adds `WorkspaceFs.writeEnv` (`writeTextFile`, `$HOME/**` cap already granted - no capability/Cargo change). `.env` stays out of `MANAGED_FILE` so reconcile still can't delete it. The earlier ADRs' "no in-app editing" was a scope deferral, not a permanent constraint |
| 2026-06-20 | Secret variables deferred; `.env` (gitignored) is the only secrets mechanism this round | Workspace files are already documented plaintext; building Bruno's encrypted secret store (OS keychain, `vars:secret`, AES fallback) is a large separate feature. `.env` at the workspace root + `{{process.env.X}}` covers "keep secrets out of git" without crypto wiring |
| 2026-06-20 | Request body is a tagged `StoredBody = {type:"json"; payload:<parsed>} \| {type:"text"; payload:string}` ON DISK + in the Settings JSON, but stays a plain `string` IN-MEMORY (`RequestNode.body`); `schemaVersion` 2->3 | User: editing a JSON body in the full-request Settings JSON was painful because it showed as an escaped `"{\n ...}"` string. Storing a json body as its PARSED value renders it as real nested JSON (no escaping) and is the data-model foundation for body-types (feature #4). Kept in-memory `body:string` (only the disk/Settings boundary converts, via `lib/workspace/body-codec.ts`) so `buildHttpRequest`, the Rust wire (`HttpRequest.body:string`), the CodeMirror Body tab, the `requestOverrides` map, dirty-compare, and ~50 fixtures are untouched - the change is two boundary calls + a codec, not a model-wide ripple. Public on-disk contract change, hence the version bump; tolerant decode keeps legacy v2 bare-string bodies loading (they migrate to v3 on next save). Only `json`/`text` variants now; `form-urlencoded`/`multipart` are feature #4. Alternative (raw-string payload, Bruno-style) rejected: it leaves the body escaped in Settings, i.e. doesn't fix the reported pain |
| 2026-06-21 | Tree CRUD (create/rename/delete/duplicate) surfaced THREE ways - row right-click `ContextMenu` + command-palette entries + rebindable shortcuts - all persisting through the existing `onTreeChange`/`writeWorkspace` reconcile seam | User directive: ops must be command-palette actions WITH keyboard shortcuts, not just buttons; the context menu is the natural per-row target-picker. No new persistence path (reuses feature #1/drag-move's `persistTree`), no on-disk format change (`serialize` regenerates paths + `order` from structure; reconcile writes new files + removes orphans/empty dirs). `radix-ui`'s `ContextMenu` (already in the installed meta pkg) - no new dep. Alternative (hover icon buttons only) rejected against the explicit palette+shortcut requirement |
| 2026-06-21 | CREATE REQUEST = promote the in-memory draft, with SAVE (`Mod+S`) writing its `*.req.json`; CREATE FOLDER writes immediately + enters inline rename | Feature #1 explicitly named tree-crud as the thing that flips draft-save from a no-op to a real disk write, so reusing the draft tab + `saveActiveRequest` precedence is the smallest change. A folder has no tab/editor to host an unsaved draft, so it persists on create and drops straight into inline rename. The draft tab's id is swapped (`draft-<n>` -> `new-<n>`) on save so the tab survives as the saved node. Alternative (direct on-disk create for requests too, no draft) rejected: it duplicates the draft machinery and loses the "edit before first save" tab |
| 2026-06-21 | Created nodes get an in-session synthetic id `new-<n>`; placement is relative to the tree selection; delete confirms only for a non-empty folder | `new-` (not `draft-`) so created nodes persist + aren't re-treated as drafts; on the next disk reload the id becomes path-based (accepted, same as drag-move). Selection-relative placement (inside folder / sibling of request / root) reuses `MoveTarget` + `locateNode` - the same vocabulary drag-move speaks. Confirm-only-for-non-empty-folder (user decision): a single request / empty folder deletes immediately (low cost, undo-able by recreate); a non-empty folder is the one destructive multi-file case worth a dialog |
| 2026-06-21 | SUPERSEDES the draft-save row above: CREATE REQUEST behaves exactly like CREATE FOLDER - writes a real node to disk IMMEDIATELY, opens+selects it, opens its tab, enters inline rename. The DRAFT SYSTEM is removed entirely | User directive: a new request should appear in the sidebar instantly and focus its rename, mirroring new folder - not live as an unsaved draft tab. To appear in the tree a request must be a real node, so the draft model lost its only purpose. Removing it (no `drafts` state, no `draft-`->`new-` id swap, no draft-save branch in `saveActiveRequest`) is a net simplification: a created request is a normal on-disk node (clean until edited, no special-casing in dirty/close/persist). The persist-edits feature's draft-specific ACs are now historical |
| 2026-06-21 | Body modes (`json`/`none`/`form`/`multipart`) encode to a STRING on the frontend; the Rust wire contract (`body: Option<String>`) is unchanged - NO `send_http_request` change. Multipart is TEXT PARTS ONLY (no file uploads) | The single highest-leverage decision: text-only form/multipart can be fully serialized in TS (urlencoded via `URLSearchParams`, RFC-7578 multipart in `body-encode.ts`), so the entire file-dialog + binary-wire + Rust `multipart::Form` surface is avoided. File-upload parts (native picker, disk read, structured wire payload) deferred as a user product call. Multipart boundary is a FIXED deterministic token (not random) because the workflow/test env forbids `Math.random`, and it keeps wire output stable + unit-testable |
| 2026-06-21 | Body data model = `bodyMode?` tag + SHARED `bodyForm?: KeyValue[]` rows beside the existing `body` string (json slot), NOT one ADT union value | User chose "preserve data on mode switch": form<->multipart share one row set (same shape, reuse `EditableKeyValueTable` with toggle), and the JSON text keeps its own `body` slot, so switching json->form->json never destroys the JSON text. A single union value would discard the other mode's data on every switch. `bodyMode`/`bodyForm` are optional (absent => json + []) so every legacy request and ~50 fixtures are untouched, and both are OMITTED from disk + the Settings JSON when at default to keep `*.req.json` diffs minimal. Threaded through the existing `RequestOverride`/`RequestPatch` (no new persist path) |
| 2026-06-21 | Each body mode AUTO-SETS its `Content-Type` (incl. json => `application/json`), an explicit user header wins; `none` and GET/DELETE set no body + no auto CT | User decision: auto-set for ALL modes, user header overrides. This is a deliberate BEHAVIOR CHANGE - existing JSON requests now send `application/json` automatically (previously the user added it by hand). The override check is case-insensitive on the resolved headers and prevents the auto header being added (exactly one CT sent). Multipart's CT carries the boundary, so its auto-set is mandatory for a valid request. Alternative (auto-set new modes only, leave json untouched) rejected by the user in favor of Postman-like consistency |
| 2026-06-21 | Pre/post request scripts execute in **QuickJS compiled to WASM** behind a `ScriptRunner` port, NOT a hand-rolled `new Function`/`eval`, a Web Worker, or `isolated-vm`. Engine = `quickjs-emscripten-core` + the **single-file BROWSER variant** `@jitl/quickjs-singlefile-browser-release-asyncify` (WASM base64-embedded in the JS), NOT the umbrella `quickjs-emscripten` default (separate `.wasm` asset) | User directive: use the recommended pattern, not a custom solution. This is a Tauri **webview** (no Node `vm`/`isolated-vm`; `new Function` shares the app realm + can reach `window`/`fetch`), so a real in-browser sandbox = QuickJS-WASM. It is exactly what Bruno (the file-based client this app mirrors) migrated to after its vm2 sandbox-escape CVEs (Sonar writeup). Real realm isolation, killable via the interrupt handler (timeout), browser+native identical. The single-file variant is MANDATORY (not just smaller): the default variant's separate `.wasm` fetch fails in the WKWebView (`CompileError ... doesn't start with '\\0asm'`) - shipped that by mistake first, it passed vitest+build but crashed the real webview. Asyncify variant because user scripts may `await`. Port/fake seam (mirrors `HttpClient`/`fs`) keeps the send-loop + helpers WASM-free in tests. Web Worker rejected (async-only + serialize across boundary, no in-place req mutation); `isolated-vm` rejected (Node-only, not webview-compatible) |
| 2026-06-21 | Scripts run **frontend-side** in `sendRequest`; the Rust transport is unchanged. `buildHttpRequest` is NOT modified - the pre-script layer produces a modified node + modified `EffectiveConfig`, all interpolation/body-encode/auth stay as-is | The send loop already lives in the frontend (`workspace-context.tsx`); Rust is a thin `reqwest` transport. Running scripts frontend-side means ONE implementation covers `npm start` (native) and `npm run dev` (browser), no Rust/serde/capability surface. Feeding pre-mutations through the existing `buildHttpRequest` (rather than editing it) means a script-set `req.setUrl("{{baseUrl}}/x")` is still interpolated/encoded/auth-applied for free, and the no-scripts path is provably behavior-preserving (regression test) |
| 2026-06-21 | `requi.setVar` **persists to disk** (into `config.variables`), writing to the **nearest scope that already defines** the name (leaf-first walk of the resolved chain) else the request's own config; a **pre** error ABORTS the send, a **post** error only logs; scripts support `async`/`await`; API = `req`/`res` (Bruno-identical) + a `requi` namespace, `getX`/`setX` methods | All user decisions. Persist-to-disk (vs an in-memory runtime store) makes chaining survive reload but rewrites `*.req.json`/`folder.json` per send (git churn, batched to one `persistTree`/run). Nearest-defining-scope write (vs always-request) updates the var "where it logically lives" at the cost of a wider blast radius (a shared `folder.json`). Pre-abort prevents sending a half-applied request; post-only-log keeps a received response visible. `requi` (not Bruno's `bru`) is on-brand while `req`/`res`+`getX`/`setX` keep pasted Bruno scripts mostly portable; method calls (not property proxies) marshal trivially across the WASM boundary. Env-block writes, a tests/assertions block, flow control, and fetch/timers are explicitly OUT (YAGNI / item #14 rejected) |
| 2026-06-23 | E2E = **Playwright against the `npm run dev` browser build wired to fakes**, NOT native `tauri-driver`/WebdriverIO against the packaged binary | macOS ships no WebDriver for an embedded WKWebView (`safaridriver` only drives Safari.app; Tauri's docs list macOS as unsupported for `tauri-driver`), so native E2E would be Linux-CI-only and undebuggable on the primary dev machine. The real Rust send/cancel/serde boundary is already covered by `cargo test` + `wiremock`; Playwright-on-fakes covers the UI flows (tree CRUD, tabs, palette, send) locally and fast. Accepted gap (per the standing todos decision): E2E exercises the fakes, not the native `reqwest`/disk/WebView boundary |
| 2026-06-25 | **Bruno import reads BOTH on-disk formats: `.bru` markup AND OpenCollection YAML** (`*.yml` + `opencollection.yml`/`folder.yml` + `environments/*.yml`), via a second pure parser `parseOpenCollection` yielding the same `ParsedBru` shape; `brunoToTree` dispatches per file extension. Adds the `yaml` dep | Real-world exports (the user's AS24/Postman-converted collections) are OpenCollection YAML, not `.bru` - 487 `.yml`, zero `.bru` - so the `.bru`-only importer imported nothing. The two formats are alternative serializations of one model, so a second parser to the SAME `ParsedBru` keeps the mapper + tree-build format-agnostic (one dispatch point, `.bru`->parseBru else parseOpenCollection). YAML headers/params/vars are `{name,value,disabled,type}` lists (vs `.bru` dict blocks), body is `{type,data}`, auth a string or object, scripts `[{type,code}]`; folder/collection config lives under a `request:` block in `folder.yml`/`opencollection.yml` (nested `opencollection.yml` in Postman-converted repos = a folder-config carrier, not a request). New dep `yaml` (^2.9) over hand-rolling a YAML parser (same "use the right small lib" stance as cmdk/dnd-kit/quickjs). Verified against all 9 real collections (389 requests) - extraction correct (method/url/headers/body/auth/`{{var}}`), no throws |
| 2026-06-25 | **Import Bruno collection = a folder picker + two pure modules** (`parseBru` block parser, `brunoToTree` file-map->tree fold), result inserted as ONE new top-level folder via the existing `onTreeChange`/`persistTree` path; NO Rust change | A Bruno collection is a DIRECTORY of `.bru` files, so the surface mirrors open-workspace's `FolderPicker` (a new `BrunoCollectionReader` port threaded loader->layout->main), NOT cURL import's paste dialog. Additive (a new sibling folder, never clobbers the open workspace) mirrors cURL import's "new node, never touch the active request" rule scaled to a subtree. Bruno `environments/<name>.bru` `vars` fold into the imported root folder's `config.environments.<name>` (ReqUI stores envs in config, ADR 2026-06-20, not env files). Reading `.bru` is `@tauri-apps/plugin-fs` from the frontend (capabilities already grant `fs:read-dir`/`fs:read-text-file` under `$HOME/**`) so the Rust transport is untouched; the import produces ordinary nodes the existing `serialize` writes (no on-disk format bump). Lenient parse (skip `tests`/`assert`/`docs`/`body:graphql`, best-effort on garbage) matches the cURL importer's tolerance for pasted real-world input. Alternative (a paste dialog for a single `.bru`) rejected: a collection is inherently multi-file + a tree, the value is importing the whole thing |
| 2026-06-23 | The **`npm run dev` browser build seeds a demo workspace** (in-memory fs + fake HTTP + noop picker + demo settings store), discriminated by `isDevBrowser()` = `import.meta.env.MODE === "development" && !isTauri()` | Two birds: it makes `npm run dev` actually interactive (something for Playwright to drive end-to-end) AND promotes the former `mockTree`/`mockConsoleLines` from dead seed (the prod loader + every test already passed `tree` explicitly, so the `WorkspaceProvider` defaults were unreachable) to the live demo seed in `src/lib/workspace/demo-seed.ts`, resolving the `mock-data.ts` cleanup smell. Gated on `MODE` not `import.meta.env.DEV` because Vitest also sets `DEV=true` - `DEV`-gating would seed the demo into jsdom tests and break the empty-state assertions; `MODE` is `"test"` under Vitest. The seed round-trips through `serialize`/`deserialize` so it can't silently drift from a shape the loader rejects. `npm start`/build/vitest paths are unchanged. Runners partition by file extension (`*.e2e.ts` = Playwright, `*.spec`/`*.test` = Vitest) so the two never collide |
