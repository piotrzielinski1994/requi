- TanStack Hotkeys split: `@tanstack/hotkeys` is the framework-agnostic core (no React hook). The React `useHotkey` + `HotkeysProvider` live in `@tanstack/react-hotkeys`. Install the adapter, not the core.
- Hotkey strings are case-sensitive in the typed union: use uppercase keys, e.g. `"Mod+K"` not `"Mod+k"`.
- Under jsdom the hotkeys lib resolves `Mod` to `Control` (test platform reports non-mac), so hotkey tests fire `{Control>}k{/Control}`, not Meta.
- ESLint react-hooks v7 false-positives: `useReactTable` trips `react-hooks/incompatible-library`; code-based TanStack route files trip `react-refresh/only-export-components`. Both scoped off in eslint.config.js for the relevant paths.
- shadcn Button keeps `react-refresh/only-export-components` as an accepted warning (canonical upstream file exports `buttonVariants` alongside the component). Lint exits 0 with warnings.
- `npm create tauri-app` can't target a non-empty dir: scaffold in temp, copy `src-tauri/` + vite/ts configs + `index.html` in, then rewrite identity (Cargo `name`, `tauri.conf.json` productName/identifier, `main.rs` `_lib` ref).
- shadcn `resizable` ships for the old `react-resizable-panels` API: it passes `direction` to `ResizablePanelGroup`, but v4 renamed the prop to `orientation` ("horizontal"|"vertical"). The generated `resizable.tsx` types are fine; fix the prop at call sites or typecheck fails.
- react-resizable-panels v4 `Panel` size props (`defaultSize`/`minSize`/`maxSize`) read a bare `number` as PIXELS, not percent. `defaultSize={20}` = 20px wide. For proportional panels pass a string with a unit: `defaultSize="20%"`. (Symptom: a panel renders a few px wide and won't expand past its px `maxSize`.)
- jsdom has no `ResizeObserver`; radix Select/Tabs + react-resizable-panels need it. A no-op `ResizeObserver` stub is installed in `src/test/setup.ts`. Also expect harmless "Not implemented: Window's scrollTo()" noise from shadcn `ScrollArea` under jsdom.
- radix `SelectValue` renders nothing until the dropdown opens (items live in an unmounted Portal), and jsdom can't open it. To assert the current value in tests, render the value as explicit `SelectTrigger` children instead of relying on `<SelectValue/>`.
- react-resizable-panels v4 persistence: pass `defaultLayout` (a `{panelId: number}` map) + `onLayoutChanged` to `Group` (our `ResizablePanelGroup`, which spreads props through). `onLayoutChanged` fires on pointer release (use it for saving, not `onLayoutChange` which fires per-move). Panels need stable string `id`s for the map keys. When a group has no saved layout, pass `defaultLayout={undefined}` and the lib falls back to each panel's `defaultSize`.
- `nvm` on this machine is shimmed to print "use `mise` instead" and does nothing; node is managed by mise (`.nvmrc` pins 24, mise has 24.17.0). In non-interactive bash, activate first: `eval "$(mise activate bash)"` then `mise exec -- <cmd>` to get node 24.
- Anything under `SettingsProvider` (the resizable groups via `useSettings`) only mounts children AFTER the async settings load resolves. Tests rendering `WorkspaceLayout`/`Main`/`Content` must wrap them in `SettingsProvider` (use `createInMemorySettingsStore()`) and use `findBy*`/`waitFor` for the first assertion.
- Domain types for the workspace tree live in `src/lib/workspace/model.ts` (so the pure `lib/workspace` layer owns them, not a component). `components/workspace/mock-data.ts` re-exports them, so existing `@/components/workspace/mock-data` type imports keep working - dependency points components -> lib, never the reverse.
- The tree's per-node config is a single `config: ConfigScope` (variables/headers/params/auth/scripts/timeout), NOT flat fields on the request. `Auth` has an `inherit` variant (the default for a request that defers to its folder). A request's `response` is optional and never persisted to disk.
- `resolveConfig(tree, requestId)` is pure: it folds the folder chain root->request. Collections (variables/headers/params) merge per-key (headers case-insensitive by name, keeping the winner's casing); scalars (auth whole-object/scripts.pre/scripts.post/timeoutMs) take the nearest-defined value. Unset falls back to a synthetic `{scopeId:"default"}` provenance (auth `{none}`, timeout `DEFAULT_TIMEOUT_MS`=30000).
- Tauri v2 `fs` plugin: a user-arbitrary read path needs both the action perms (`fs:allow-read-dir`/`-read-text-file`/`-stat`) AND an `fs:scope` allow glob in the capability. We use `$HOME/**` (workspace must live under home - documented constraint). `readDir` is NOT recursive - recurse manually building relative paths.
- A `setState` called synchronously (not in a callback/promise) inside a `useEffect` trips eslint `react-hooks` ("cascading renders"). For a value derivable at render (e.g. empty-state when no `workspacePath`), compute it in the `useState` initializer instead of setting it in an effect; keep the effect for the genuinely async branch only.
- `@tanstack/hotkeys` `validateHotkey(s)` returns `{valid, warnings, errors}` and is too lenient for override validation: an unknown key (e.g. `"bogus!!"`) is `valid:true` with only a *warning* `"Unknown key: '...'"`. To reject garbage in `safeNormalize`, treat a warning containing `"Unknown key"` as invalid (not just `!valid`). Empty string is the only thing that flips `valid:false`.
- `useHotkeys`/`useHotkey` from `@tanstack/react-hotkeys` do NOT require `HotkeysProvider` - `useHotkeysContext()` returns null when absent and the hook just falls back to default options. So components using `useActionHotkeys` test fine under `SettingsProvider` alone (no need to add `HotkeysProvider` to every test).
- `UseHotkeyDefinition.hotkey` is typed `RegisterableHotkey = Hotkey | RawHotkey` where `Hotkey` is a closed literal union (no `(string & {})` escape). A runtime-validated `string` binding must be cast `as Hotkey` at the `useHotkeys` boundary. Same closed-union reason the override map stores plain `string`, not `Hotkey`.
- The recorder (`useHotkeyRecorder`) reserves `Escape` for cancel, so `Escape` cannot be assigned to an action via the UI - it stays usable only as a registry default (we use it for `close-settings`). Side effect: pressing `Esc` while recording a binding both cancels the recording AND fires `close-settings` (closes the settings tab). Accepted.
- Settings is NOT a route - it's an in-app tab tracked in `WorkspaceProvider` (`isSettingsOpen`/`isSettingsActive` + `openSettings`/`closeSettings`), rendered in the content area by `Content` when active. Making it a `/settings` route caused two bugs: (1) `navigate()` built browser history so `Alt+ArrowLeft/Right` walked it instead of cycling request tabs; (2) leaving `/` unmounted `WorkspaceProvider`, wiping open tabs + expanded folders. A WebView/jsdom history nav from a route can't be reliably `preventDefault`-ed away - the fix is to not create the history entry at all. The settings hotkeys (`open/close-settings`) now live in `Main` (always mounted) operating on workspace context, not in a router-level component.
- Selecting/activating a request (`selectNode`, `setActiveRequest`, next/prev-request) clears `isSettingsActive` so the request view replaces settings, but leaves the settings tab open (`isSettingsOpen` stays true). Closing settings (× or `Esc`) clears both.
- The workspace shell (sidebar/content/console) always mounts, including the no-workspace case (`WorkspaceLoader` empty branch renders `WorkspaceProvider tree={[]}`). The "No workspace" hint lives in `SidebarTree` (empty tree). Keep that phrase out of the empty-state console line, else `findByText(/no workspace/i)` matches two nodes.
- HISTORICAL (SUPERSEDED by tree-crud): new-request once opened an in-memory `draft-<n>` node merged into `requestsById`, never persisted. tree-crud removed the draft system - `newRequest` now inserts a real on-disk node immediately (see the CREATE REQUEST = CREATE FOLDER entry).
- `open-workspace` uses a `FolderPicker` port (`src/lib/workspace/folder-picker.ts`): `createTauriFolderPicker()` wraps `@tauri-apps/plugin-dialog` `open({directory:true})` and catches -> null when Tauri absent; `createNoopFolderPicker()` for tests/browser. The picker is created in `routes/index.tsx` (`useState(createTauriFolderPicker)`) and threaded as a prop `WorkspaceLoader -> WorkspaceLayout -> Main` (mirrors how `fs` is passed). `Main` registers `open-workspace`; on a non-null pick it calls `saveWorkspacePath(path)` and the loader's `workspacePath` effect reloads. Required Tauri wiring: `tauri-plugin-dialog` in Cargo + `tauri_plugin_dialog::init()` in lib.rs + `dialog:default`/`dialog:allow-open` in capabilities + `@tauri-apps/plugin-dialog` npm dep.
- Sidebar hide gate lives in `WorkspaceLayout` (renders only the content panel when `settings.sidebarHidden`), NOT in `SidebarTree`/`Sidebar` - keep `SidebarTree` settings-free so its unit tests don't need a `SettingsProvider`. Tests that want to exercise the hide must render through `WorkspaceLayout`, not `SidebarTree` standalone.
- Default tab-switch shortcuts are `Ctrl+Tab` / `Ctrl+Shift+Tab` (NOT `Alt`+arrows or `Cmd+W`). Reason: the user's machine runs Karabiner-Elements (config in the `config` repo, `~/.config/karabiner/karabiner.json`) which remaps at the OS level BEFORE the WebView: `Cmd+W`->`e`, `Option+Left/Right` (= `Alt`+arrows) -> bare arrows / `a`/`e`, and `Cmd|Option+Tab` are taken. `Ctrl+Tab` / `Ctrl+Shift+Tab` are the only tab-ish combos Karabiner leaves untouched. Gotcha when debugging hotkeys on macOS: synthetic events from AppleScript `System Events` / CGEvent BYPASS Karabiner, so a shortcut can look like it "works" under scripted testing yet be eaten by Karabiner on a real physical keypress. Always consider an OS-level remapper (Karabiner, Hammerspoon, BetterTouchTool) before concluding an in-app hotkey is broken.
- react-resizable-panels registers a global `keydown` handler on each separator for ARROW-key resize. App hotkeys that used arrows (the old `Alt+ArrowLeft/Right`) in a test mounting a `ResizablePanelGroup` made that handler throw `Previous layout not found for panel index N` (jsdom panels measure 0x0, layout never commits) as an uncaught error -> vitest exits 1 despite passing assertions. Switching the default to `Ctrl+Tab` removed the trigger entirely (Tab doesn't hit the arrow-resize path), so the `src/test/setup.ts` error-swallow workaround was deleted. If a future shortcut uses arrow keys near a panel group, the noise (and the need to filter it) returns.
- `SettingsProvider` is mounted at the root route (`__root.tsx`), not per-page, so `/settings` (outside the home route) can read/write settings. It renders children only after the async load resolves - this gates EVERY route. The `settingsStore` instance is created once in `RootLayout` via `useState(createTauriSettingsStore)`.
- cmdk (shadcn `command`) filters each item on its `value` prop (defaults to rendered text content). Command-palette rows render both the action name AND the shortcut glyph, so `value` MUST be set explicitly to `action.name` - else typing would match the glyphs too. cmdk's `CommandInput` renders with `role="combobox"` (not `textbox`) - query it that way in tests. cmdk calls `scrollIntoView` on highlight move; jsdom lacks it, so `src/test/setup.ts` stubs a no-op `Element.prototype.scrollIntoView` (mirrors the `ResizeObserver` stub).
- Adding a new action to `SHORTCUT_ACTIONS` breaks any test that used that action's new default hotkey as a "free/unused" combo example. `Mod+K` was free until the command palette claimed it; the shortcuts-section rebind tests had to switch their free-combo example to `Mod+Y`. When picking a new default hotkey, grep the test suite for that combo first.
- The command palette reuses `Main`'s single `handlers` map for BOTH `useActionHotkeys` registration AND the palette's `commands` list (`SHORTCUT_ACTIONS` minus `open-command-palette` × effective bindings × `handlers`). A palette command does exactly what its shortcut does - no second source of truth. Palette open-state is local `useState` in `Main` (pure UI ephemeral state), not in `WorkspaceProvider`. `open-command-palette` is just another registry action (default `Mod+K`), so it's rebindable/conflict-checked/persisted with zero special-casing.
- `useActionHotkeys` must NOT pass a global `{ ignoreInputs: true }` to `useHotkeys` - that suppressed EVERY action whenever focus was in an input or the CodeMirror config/`.env` editor (so `Mod+K`/`Mod+Enter`/`Mod+S` died inside the Settings tab). `@tanstack/hotkeys` already picks per-hotkey via `getDefaultIgnoreInputs`: Mod/Ctrl combos and `Escape` fire even in inputs; bare keys stay suppressed while typing. Omit the option and let the lib decide. jsdom can't exercise the editor path (it leaves `HTMLElement.isContentEditable` `undefined`, so `isInputElement` treats `.cm-content` as non-input) - cover the rule with a real `<input>` + a bare-key binding instead.
- `Mod+S` saves whatever editor is active via a saver-registration channel on `WorkspaceProvider`: the mounted editor form (`ConfigEditorForm`, `EnvEditorForm`) calls `registerEditorSaver(() => save(currentDraft))` in a `useEffect` (deps include the draft so the closure stays fresh) and clears it on unmount; the `save-active-editor` action handler in `Main` calls `saveActiveEditor()` which invokes the registered saver (no-op if none). Draft text lives local to each editor, so a global hotkey can't read it directly - the ref indirection is why. Testing tip: rendering full `Main` inside a `ResizablePanelGroup` makes sub-tab clicks flaky under jsdom (pointer interception) - drive `saveActiveEditor` from a tiny harness button instead of the real hotkey (the hotkey->handler binding is already covered by `use-action-hotkeys` tests).
- `Settings.shortcuts` stays a single model field, but the Tauri adapter (`createTauriSettingsStore`) splits persistence across TWO store files: everything except `shortcuts` -> `settings.json`, `shortcuts` -> `keymap.json` (so a keymap is device-syncable independently of device-local UI state). On `save` the adapter writes `shortcuts: {}` into settings.json to avoid duplicating the keymap; on `load` it merges settings.json then overlays keymap.json's shortcuts. The split lives ONLY in the adapter - the model/`mergeSettings`/context/in-memory-store are file-agnostic, so tests don't change. `store:default` is plugin-scoped (not per-file), so the second LazyStore needs no capability change.
- CodeMirror 6 (`@uiw/react-codemirror`) mounts fine under jsdom with NO extra stubs - `.cm-editor`/`.cm-content` render, `.cm-content` carries `role="textbox"` + `contenteditable="true"` + `data-language="json"` (from `@codemirror/lang-json`), token `<span>`s appear, and `EditorView.findFromDOM(el)` reaches the live view. But jsdom can't simulate live keystroke editing into the contentEditable surface - to test edits, `view.dispatch({changes})` and assert `onChange` fired; to test auto-close-brackets WIRING (not behavior), assert `view.state.facet(EditorView.inputHandler).length >= 1` (`json()` alone = 0 handlers; adding `closeBrackets()` = 1, so the assertion fails if closeBrackets is removed - honest, not tautological).
- Dev server port is hardcoded 1430 (Vite `server.port` in vite.config.ts + `build.devUrl` in tauri.conf.json - both must match or the Tauri webview loads a blank page). Changed from the Tauri-template default 1420.
- CM6 `@codemirror/lint` dispatches diagnostics asynchronously: `forceLinting(view)` runs the lint source synchronously but pushes results through `Promise.resolve(source(view)).then(...)` (a microtask), so `diagnosticCount(view.state)` is still 0 on the line right after `forceLinting`. In tests, make the test `async` and `await Promise.resolve()` (one microtask flush) before asserting the count. Body editor uses `linter(jsonParseLinter())` + `lintGutter()` (both ship with `@uiw/react-codemirror`'s peers; `jsonParseLinter` from `@codemirror/lang-json`) - syntax/parse errors only, NOT JSON-Schema type validation.
- `@uiw/react-codemirror` defaults `theme` to `'light'` when the prop is OMITTED (not "no theme"). The light theme injects a SOLID WHITE editor-wrapper background that overrides a custom `EditorView.theme` passed via `extensions`, so the editor renders white even with a dark custom theme present. To use ONLY a custom theme, pass `theme="none"` explicitly (supported values: `'light'|'dark'|'none'|Extension`). For the body editor we want it to look identical to the surrounding pane, so the custom Darcula theme sets `backgroundColor: "transparent"` on `&`/gutters/activeLine (inherits the pane) rather than a solid color. GOTCHA when testing this in jsdom: CM6 themes are global `StyleModule` rules injected into `<style>` tags in `document.head` and deduped across the whole test run - rendering a default-light CM in ANY earlier test leaves its `.ͼN {background-color:#fff}` rule in the head, polluting later tests; the regression test must render ONLY BodyEditor (own file) and match the bare-class wrapper rule (`{...background-color:#fff}` NOT under a `.cm-*` selector), since `.cm-snippetField`/`.cm-textfield` carry legit unrelated whites.
- Body editor theme = JetBrains Darcula, built WITHOUT a new dep: `@uiw/codemirror-themes`/`createTheme` is NOT re-exported by `@uiw/react-codemirror`, so instead compose `EditorView.theme({...},{dark:true})` for chrome (bg/gutter/selection/caret) + `syntaxHighlighting(HighlightStyle.define([...]))` from `@codemirror/language` mapping `@lezer/highlight` tags to hex (string `#6a8759`, number `#6897bb`, keyword/bool/null `#cc7832`, propertyName `#9876aa`, bg `#2b2b2b`, fg `#a9b7c6`). All those pkgs ship as peers of the wrapper - no install. Don't also pass the `theme=` prop when a custom theme extension is in `extensions` (double-theming).
- `@testing-library/jest-dom` `toHaveTextContent` normalizes whitespace by default, so a multiline string (e.g. JSON body with `\n`) won't match the rendered text node. Pass `{ normalizeWhitespace: false }` as the 2nd arg to assert exact multiline content.
- Body editing is in-memory via `WorkspaceProvider.bodyOverrides: Map<id,string>` + `setRequestBody(id, body)`; `requestsById` applies overrides as `{...base, body}` (never mutates tree/draft nodes). `closeRequest` deletes that id's override and `closeAllRequests` clears the whole map, so a reopened tree request reverts to its original on-disk body (drafts are disposed entirely). No disk write - persistence is a separate unbuilt feature. The Body `TabsContent` renders `<BodyEditor key={request.id}>` so switching tabs resets the CM doc cleanly instead of diffing.
- Open request tabs persist via `Settings.openRequestIds` + `activeRequestId`. `WorkspaceProvider` is kept settings-free (DI): it takes `initialOpenRequestIds` (restored, filtered to ids present in the current tree) + an `onTabsChange(ids, activeId)` callback; `WorkspaceLoader` wires both to `settings`/`saveOpenTabs`. The persist effect skips the first render (an `isFirstTabsRender` ref) so restoring doesn't immediately re-save, and strips `new-` ids (a freshly-created node's in-session synthetic id won't match a path-based id after reload, same as drag-move). GOTCHA: the callback must be held in a ref (assigned in an effect, NOT during render - eslint `react-hooks` forbids ref-writes in render) and kept OUT of the effect deps; including `onTabsChange` (whose identity changes every settings update) in the deps caused an infinite setState loop ("Maximum update depth exceeded"). `mergeSettings` nulls `activeRequestId` if it isn't among `openRequestIds`.
- DnD via `@dnd-kit/*` (core+sortable+utilities). Tabs use `SortableContext` (horizontal) + `arrayMove` -> `reorderRequests` (pure permutation guard on `openRequestIds`). Sidebar is NOT sortable (tree is nested): each `TreeRow` is `useDraggable`+`useDroppable`, a pointer projection over the row's vertical thirds picks `before`/`after`/`inside` (inside only over a folder, auto-expanding it), surfaced via a `TreeDndProvider` context (activeId + indicator) so rows render an insertion line / folder ring without prop-drilling. jsdom CANNOT simulate dnd-kit pointer gestures - test the pure layer (`moveNode`, `dropTarget`, `planReconcile`, `order` round-trip) + the indicator render contract (seed `TreeDndProvider` directly), never raw drag events. When spreading `useDraggable` `attributes`/`listeners` onto a row, spread them BEFORE your explicit `role`/`tabIndex`/`aria-*` or dnd-kit's defaults (role=button, tabIndex=0) overwrite yours and break the `treeitem` queries (TS2783 flags it).
- `dropTarget(tree, dragId, overId, position)` must compensate for `moveNode`'s POST-removal index semantics: when dragging within the same parent and the dragged node sits before the drop point, subtract 1 from the raw index (else a downward drag lands one slot too far). Cross-parent and `inside` drops skip the shift. The sidebar's `handleDragEnd` also early-returns when the resolved target equals the node's current `{parentId,index}` (no spurious disk write on a same-spot drop).
- Workspace is now WRITABLE on a sidebar move (the first writer - `serialize()` was previously unused). `WorkspaceFs.writeWorkspace(rootPath, files)` (Tauri adapter): re-reads current managed FileMap, `planReconcile` -> mkdir(recursive)+writeTextFile the write set, `remove` managed orphans, then `remove` now-empty managed dirs deepest-first (`emptyDirsAfterRemoval`). Removal is regex-gated to managed files only (`folder.json`/`*.req.json`/`requi.workspace.json`) so user files are never touched; `remove` on a non-empty dir throws and is swallowed (`.catch`). Needs `fs:allow-write-text-file`/`-mkdir`/`-remove` capabilities. `tree` + `consoleLines` are now `WorkspaceProvider` STATE (were props); `moveNode(dragId,target)` applies the pure reducer + calls an injected `onTreeChange(tree)` whose `{ok:false}` appends a `[workspace] failed to persist move: ...` console line (in-memory is session source of truth; a failed write never rolls back). Loader keys `WorkspaceProvider` by `workspacePath` so a new workspace remounts fresh (lifted state would otherwise not reseed on prop change), and threads the manifest name through `serialize(tree, name)` so a move doesn't rename the workspace to the default.
- On-disk format is schemaVersion 2: every `folder.json`/`*.req.json` carries a numeric `order` (sibling array index). `serialize` PRESERVES caller array order (no longer alpha-sorts) and writes `order`; `deserialize` sorts siblings by `order` ascending, falling back to the old folders-first-then-name rule for nodes missing `order` (legacy v1 files still load - tolerant). Without this, sibling reorder silently reverted on reload.
- HTTP send goes through a Rust `send_http_request` async command (`reqwest`, `default-features=false`+`rustls-tls` to dodge system OpenSSL). The webview is CORS-bound, so real cross-origin requests MUST be a Tauri command, not webview `fetch`. The command needs NO new capability/scope (commands are allowed by default; reqwest opens sockets from the Rust process, not the webview). An `async fn` Tauri command runs on Tauri's tokio runtime - no manual `async_runtime` wiring. First `cargo build` after adding reqwest is slow (~30s, big transitive tree). Wire shape is camelCase via `#[serde(rename_all="camelCase")]` (`timeoutMs`/`timeMs`/`sizeBytes`); body is `Option<String>` and attached only `if let Some(..)`.
- The HTTP layer follows the same port/adapter seam as `fs`/`folder-picker`: `HttpClient` interface (`src/lib/http/model.ts`), `createTauriHttpClient` (the only file importing `@tauri-apps/api/core` for this), `createFakeHttpClient` for browser/`npm run dev`. Threaded as a `WorkspaceProvider` prop (loader -> provider), NOT down through layout/main, since only the provider's `sendRequest` needs it. The provider holds it in a ref + falls back to the fake when absent so `sendRequest` is always callable.
- `bodyOverrides: Map<id,string>` was generalized to `requestOverrides: Map<id, Partial<{url,method,body}>>` so url/method edits share the body-edit semantics (per-id, cleared on close/closeAll, revert on reopen). `setRequestBody/Url/Method` are thin `mergeOverride` wrappers; `requestsById` merges `{...base, ...override}`. The pre-existing `body-override-context.test.tsx` is the regression net - it stayed green unchanged, proving the refactor is behavior-preserving.
- `sendRequest`'s async `.then` MUST guard `if (!responseStates.has(id)) return current` before writing the result: a request can be closed mid-flight, and an unguarded write resurrects a stale `responseStates` entry for a now-closed id. Per-request send state lives in `responseStates: Map<id, ResponseState>`; `responseState(id)` returns `{status:"idle"}` when absent. Double-send is blocked by checking `status==="sending"` before dispatch.
- Response filter is a hand-rolled JSONPath subset (`$`, `.key`, `[index]`) in `src/lib/http/filter.ts`, returning an ADT (`{ok:true,text}` / `{ok:false}`) - no throw, no dep. Empty/`$` path -> full body (but empty body -> not-ok). A matched string scalar is shown RAW (no quotes); numbers/bools `JSON.stringify`'d; objects/arrays pretty-printed at 2 spaces.
- CM6 Darcula theme is shared between the body editor and the response viewer via `src/components/workspace/editor-theme.ts` (`darculaChrome`, `darculaHighlight`, `jsonViewerExtensions`, `emptyTolerantJsonLinter`). The response body renders in a READ-ONLY `JsonViewer` (CodeMirror, `editable={false}`, no line numbers/fold gutter) that prettifies parseable JSON (`JSON.stringify(JSON.parse(x),null,2)`, falls back to raw on parse error) so the response is prettified + syntax-colored in the same scheme as the request body.
- `jsonParseLinter()` flags an EMPTY document as "Unexpected EOF" - an empty request body is a valid state, so wrap it in `emptyTolerantJsonLinter()` (returns `[]` when `doc.trim()===""`). Bug introduced when the body editor first shipped; surfaced as a red error marker on a blank body.
- CM6 renders body text as MANY tokenized `<span>`s, so `getByText`/`findByText` on a contiguous multi-token substring (e.g. `"foo": "bar"`) won't match in the response viewer. Read the live doc instead: `EditorView.findFromDOM(document.querySelector(".cm-editor")).state.doc.toString()` (mirrors body-editor.test.tsx). A SINGLE-token substring like a bare property name (`access_token`) still matches via `getByText` because it's one span.
- The URL bar field must stay EDITABLE yet keep `{{var}}`/`:param` token coloring. A plain `<input>` can't render colored spans, so layer a `pointer-events-none` absolutely-positioned colored highlight div (same split regex as the old read-only `UrlDisplay`) BEHIND a transparent-text input (`text-transparent caret-foreground`). The input still exposes `role=textbox`/`aria-label="URL"` for tests and `toHaveValue`. Enter in the field sends. Don't swap it for a shadcn `<Input>` (loses the coloring - regression).
- Per-folder/per-request VARIABLES already existed (`config.variables`, folded root->leaf by `resolveConfig`, `{{var}}`-substituted in `buildHttpRequest`) BEFORE the environments feature. The "variables & environments" work added: environments, `.env`, recursive interpolation, and auth/body interpolation coverage - NOT per-folder/request vars. Check what's already there before scoping a "Bruno-like vars" request.
- ENVIRONMENTS live inside the folder/request `config` as `config.environments: {envName: {var: val}}` (user directive), NOT in dedicated env files - so they round-trip through the existing opaque-`config` serialize/deserialize with ZERO disk-format change (no schemaVersion bump, no new loader). The env-name list shown in the selector is DERIVED from the tree (`listEnvironmentNames`), never stored. Only `.env` is a dedicated file.

- jsdom GOTCHA testing Radix `ScrollArea`: with an auto-hide `type` (`hover`, or the Radix default) the scrollbar slot NEVER mounts under jsdom - Radix needs overflow detection (a real layout / working ResizeObserver; ours is a no-op stub in `test/setup.ts`), so `querySelector('[data-slot="scroll-area-scrollbar"]')` returns null against ANY implementation (a wrong-reason RED: "expected null not to be null" passes/fails regardless of the class change). Only `type="always"` mounts the scrollbar element; the THUMB element (`scroll-area-thumb`) never mounts in jsdom at all (needs a measured size). So a scroll-area unit test must (a) render with `type="always"` to assert the bar's classes (`w-1.5`, not `w-2.5`), and (b) read the thumb className + the `index.css` rules from the source files on disk (`readFileSync`) - jsdom also mocks CSS imports to empty, so `import css from "...?raw"` returns `""` (vite `?raw` is intercepted by vitest's css handling). Reading repo files needs node types: a file-local `/// <reference types="node" />` keeps `node:fs`/`process` out of the app's no-node-types tsconfig (`tsconfig.json` `types: ["vitest/globals","@testing-library/jest-dom"]` deliberately excludes node). Production stays `type="hover"`; the test override is local.
- BRUNO IMPORT (feature #10): a Bruno collection is a DIRECTORY of `.bru` files, so the surface mirrors OPEN-WORKSPACE's `FolderPicker` (a new `BrunoCollectionReader` port `{ pick(): Promise<{name,files}|null> }` threaded loader -> layout -> main exactly like `picker`), NOT cURL import's paste dialog. Two PURE React-free modules carry the logic, unit-tested directly: `src/lib/bruno/parse-bru.ts` `parseBru(text): ParsedBru` (TOTAL - never throws; empty/garbage -> best-effort record) + `src/lib/bruno/bruno-to-tree.ts` `brunoToTree(files: BrunoFileMap, fallbackName): TreeNode[]` (returns ONE root `FolderNode` wrapping the collection, mirroring `disk-format.deserialize`'s prefix-based level build). `.bru` is a brace-block language (`name { ... }` / `name:sub { ... }`): `splitBlocks` BRACE-COUNTS (depth +=/-=) so a nested `{...}` JSON body isn't truncated at the first inner `}` (the one subtle bit - explicit nested-brace test). Dictionary blocks (`key: value`, leading `~key` = `enabled:false`) cover meta/method/headers/params:query/auth:*/vars; text blocks (dedented inner) cover body*/script:*. Active body = the method block's `body: <selector>` picks among multiple body blocks, else first present, `none`/absent -> empty. `body:text`/`body:xml` go to the json SLOT (verbatim, `bodyMode` left default) since `BodyMode` has no text/xml variant; only form-urlencoded/multipart-form set `bodyMode`. Auth: `auth:bearer{token}`/`auth:basic{user,pass}` creds block wins, else method-block `auth: none`/`inherit` selector. `tests`/`assert`/`docs`/`body:graphql` parsed-then-skipped (lenient, never fatal). `environments/<env>.bru` `vars` fold into the ROOT folder's `config.environments.<env>` (ReqUI stores envs in config, not env files - ADR 2026-06-20), and that dir is excluded from the request walk so an env file is never a request node. `bruno.json` `name` names the root, else `collection.bru` meta, else the picked dir base name. Context `importBruno(files, name)` GUARDS `root.children.length === 0` (empty collection -> no-op, no persist), overrides `brunoToTree`'s `bruno-<n>` root id with a synthetic `new-<n>` (so the open-tabs `!id.startsWith("new-")` persist filter + reload-reid convention holds), inserts ADDITIVELY at workspace root (`insertNode(tree, null, tree.length, folder)`, existing nodes untouched - same "never clobber the active request" rule as cURL import scaled to a subtree), persists via the existing `onTreeChange`/`persistTree`. NO Rust change - reading `.bru` is `@tauri-apps/plugin-fs` `readDir`/`readTextFile` from the frontend (capabilities already grant `fs:read-dir`/`fs:read-text-file` under `$HOME/**`); no on-disk format bump (import produces ordinary nodes `serialize` writes). The reader (native fs) is the one untested seam, like `tauri-fs`/`folder-picker`; all logic lives in the pure mapper the FAKE reader feeds in the integration test. [[import-export]]

YAML shapes differ from `.bru` dict blocks: `info:{name}`, `http:{method,url,headers:[{name,value,disabled}],params:[{name,value,type:query|path}],body:{type,data},auth}`, `request:{variables:[{name,value}],scripts}` (folder/collection config under a `request:` block), `runtime:{scripts:[{type:before-request|after-response,code}]}`. body `{type:json|text|xml,data}` -> verbatim `body`; `{type:form-urlencoded|multipart-form,data:[{name,value}]}` -> `bodyMode`+`bodyForm`. auth = STRING (`inherit`/`none`) OR OBJECT (`{type:bearer,token}`/`{type:basic,username,password}`). params: only `type:query`/absent kept, `type:path` dropped (URL-template part). `disabled:true` -> `enabled:false`. parseOpenCollection is TOTAL: `parseYaml` wrapped in try/catch, non-object doc -> empty best-effort record. - BRUNO query params are MIRRORED in both `http.url` (`?culture={{CULTURE}}`) AND the `params:query` block (same `culture` key) - importing both duplicated the param (`?culture=x&culture=x`) because ReqUI's `appendParams` (build-request.ts) APPENDS resolved params onto the url's existing query string (addal not mirror). Fix in `parseOpenCollection.toQueryParams(value, url)`: drop any param row whose key is already a key in the url's `?a=&b=` query string (the url query wins). `urlQueryKeys(url)` splits `url.split("?")[1]` on `&` then `=`. The `.bru` path doesn't hit this (its `params:query` block is the sole source; bru urls rarely inline the query), so the dedup lives only in the YAML parser. NOTE: a SEPARATE non-bug also visible in that 400 response - `{{process.env.CULTURE}}` left verbatim - is correct interpolation behavior: the collection references `process.env.CULTURE` but the workspace `.env` has no CULTURE, so the token stays literal (unknown-token passthrough, ADR 2026-06-20). Not an import bug; the user must set CULTURE in `.env`.
- BRUNO IMPORT merges the collection's own `.env` into the workspace `.env`: a Bruno collection ships a root `.env` (e.g. as24/.env with CULTURE=en-CA, BEARER_TOKEN=...) that feeds `{{process.env.X}}`. The importer ignored it -> imported requests left `{{process.env.CULTURE}}` literal -> 400s. Fix: (1) reader regex now also captures root `.env` into `files[".env"]` (alongside `.bru`/`.yml`/`bruno.json`); (2) `importBruno` (workspace-context) reads `files[".env"]` and calls `saveEnv(mergeDotenv(envText, collectionEnv))`. New pure `mergeDotenv(existing, incoming)` in environment.ts folds each incoming key via `setDotenvValue` (existing formatting/comments survive, existing-only keys kept, INCOMING wins on a clash - the imported collection is authoritative for the keys it ships). `saveEnv` is a `const` declared LATER in the same provider `useMemo` than `importBruno`, but the closure reference resolves at call-time (runtime, post-init) so no TDZ/no-use-before-define issue. NOTE: Bruno collections also nest per-folder `.env` files sometimes, but only the collection ROOT `.env` maps to `process.env` (matches Bruno); nested ones are read but unused.
- SCRIPT `requi.getVar`/`bru.getVar` now INTERPOLATES the returned value (Bruno returns RESOLVED values, not raw `{{...}}`). A Bruno collection pre-script does `const culture = bru.getCollectionVar("CULTURE"); culture.split('-')[1].toLowerCase()` - getVar was returning the raw config value `{{process.env.CULTURE}}` (not `en-CA`), so `.split('-')[1]` was `undefined` -> `TypeError: cannot read property 'toLowerCase' of undefined`, aborting the pre-script. Fix in `script-context.ts buildScriptApi`: build an `allVars` map (effective.variables values + runtimeVars overlay) and run the getVar result through `interpolate(raw, allVars, ctx.processEnv)` - so `{{process.env.X}}` + cross-var `{{other}}` references resolve before reaching the script (cycle-guarded/unknown-passthrough free via the shared interpolate). Covers both `requi.getVar` and `bru.getVar` (the `bru` shim bridges to the same `requi.getVar` path). NOTE: `setVar` still stores the raw value the script passes (no re-interpolation on write); only reads resolve. This is the script-side mirror of buildHttpRequest already interpolating url/headers - previously only the wire request resolved vars, scripts saw raw tokens.
- SCRIPT EDITOR linter (`script-eslint.ts` `globalsFor`) must include `bru` (the Bruno-compat alias of `requi`) in BOTH stages' `no-undef` globals - else an imported collection's `bru.setVar(...)` lights up red in the editor even though it runs fine (the runtime shim defines `bru`). - BRUNO IMPORT .env is collected at ANY DEPTH, not just collection root: the user often picks a PARENT dir holding several collections (`collections/` with `as24/`, `mbu/`...), so the `.env` they need lives at `as24/.env`, not `./.env`. Works for both a single collection (one root `.env`) and a parent-of-collections (one `.env` per child, all merged).
